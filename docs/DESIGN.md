# 設計

## 1. 設計方針

本設計では、`観測` `記録` `学習` `変換` を分離する。
マイコン側は学習用データを確実に収集する責務を持ち、PC 側はモデル作成と `.espdl` 化を担当する。

また、`顔が見えない` ことと `うつ伏せである` ことは厳密には同一ではないため、姿勢分類モデルは `顔が検出されたサンプル` のみを対象とする。
顔未検出は運用上の補助判定として保持し、分類モデルの責務に含めない。

## 2. 全体構成

### 2.1 マイコン側の責務

- カメラ映像を取得する
- 顔検出と特徴点抽出を行う
- Web 画面に映像と検出結果を表示する
- 人手ラベル付きで数値データを保存する
- CSV のエクスポートとリセットを提供する

### 2.2 PC 側の責務

- CSV を検証して不正データを検出する
- 学習用特徴量に前処理する
- データを分割してモデルを学習する
- モデルを評価する
- 評価済みモデルを `.espdl` 化する

## 3. データフロー

1. マイコンがカメラフレームを取得する
2. 顔検出と特徴点抽出を行う
3. Web 画面にフレームと検出状態を表示する
4. 人間が `うつ伏せ` または `非うつ伏せ` を選択して保存する
5. マイコンが数値特徴量を CSV に保存する
6. PC 側で CSV を集約して検証する
7. 前処理後に学習用データセットを生成する
8. モデルを学習し、評価する
9. 採用モデルを `.espdl` へ変換する

## 4. マイコン側設計

### 4.1 コンポーネント

- `camera_service`
  カメラ初期化、フレーム取得、解像度設定を行う
- `face_service`
  顔検出、顔枠取得、特徴点取得を行う
- `feature_service`
  正規化、可視フラグ設定、保存用構造体生成を行う
- `storage_service`
  SD カードへの CSV 追記、エクスポート、リセットを行う
- `web_service`
  HTTP エンドポイントと Web 画面を提供する
- `app_controller`
  各サービスを束ね、保存条件とエラー制御を行う

### 4.2 保存時の責務分担

- フレーム取得は `camera_service`
- 顔検出結果の最新化は `face_service`
- 画面表示用の矩形情報は `web_service`
- 保存時に現在の検出結果をスナップショット化するのは `app_controller`
- CSV 追記は `storage_service`

### 4.3 保存ルール

- 1回の保存操作で1行だけ追加する
- 顔が複数検出された場合は保存拒否とする
- 顔未検出時は特徴点を欠損で埋めて記録する
- 保存成否は画面に即時反映する

## 5. PC 側設計

### 5.1 スクリプト分割

- `validate_dataset`
  列欠落、型不整合、範囲外値、欠損比率を検査する
- `build_features`
  正規化、番兵値処理、学習入力生成を行う
- `split_dataset`
  セッション単位または被験者単位で分割する
- `train_model`
  初期分類モデルを学習する
- `evaluate_model`
  指標計算と誤分類分析を行う
- `export_espdl`
  採用モデルを `.espdl` 形式へ変換する

### 5.2 初期モデル設計

初期モデルは固定次元入力の軽量全結合分類器とする。
初期入力は以下の `20` 次元を採用する。

- 5点座標 `10` 次元
- 可視フラグ `5` 次元
- 顔枠 `4` 次元
- 顔検出信頼度 `1` 次元

初期層構成は以下を採用する。

- 入力 `20`
- 全結合 `32`
- 活性化関数
- 全結合 `16`
- 活性化関数
- 出力 `1`

この構成で成立しない場合のみ、派生特徴量の追加または時系列化を検討する。

## 6. 特徴量設計

### 6.1 基本特徴量

- `left_eye_x`
- `left_eye_y`
- `right_eye_x`
- `right_eye_y`
- `nose_x`
- `nose_y`
- `left_mouth_x`
- `left_mouth_y`
- `right_mouth_x`
- `right_mouth_y`

### 6.2 補助特徴量

- `left_eye_visible`
- `right_eye_visible`
- `nose_visible`
- `left_mouth_visible`
- `right_mouth_visible`
- `bbox_x`
- `bbox_y`
- `bbox_w`
- `bbox_h`
- `face_score`

### 6.3 除外条件

- `bbox_w <= 0`
- `bbox_h <= 0`
- 必須列欠落
- ラベル欠落
- 複数顔

## 7. データ品質設計

- 画像を保存しないため、収集時点のラベル品質を最優先で管理する
- ラベル操作は誤押下しにくい画面配置とする
- セッション単位で収集メモを残せるようにする
- 非うつ伏せには仰向け、横向き、部分遮蔽、暗所を含める

## 8. 評価設計

- 分割単位はセッションまたは被験者とする
- 主指標は `うつ伏せ` 再現率とする
- 副指標は `うつ伏せ` 適合率、全体正解率、混同行列とする
- 誤分類サンプルは特徴量分布で分析する

## 9. `.espdl` 化の設計

- 学習時点で入力次元と前処理を固定する
- 採用モデルは軽量構成を維持する
- 学習済みモデルの保存形式、変換コマンド、出力ファイル名を固定する
- 変換後の `.espdl` はマイコン側で読める配置にする

## 10. エラー処理設計

- 保存失敗時は CSV に中途半端な行を書かない
- SD カード異常時は保存機能を停止し、画面に状態表示する
- Wi-Fi 断時でも収集済みデータを失わない
- 変換失敗時は採用モデルを更新しない
